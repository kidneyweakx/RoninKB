## 1. Theme

**Fuse the daemon and the UI into one macOS app.** Press `⌘⇧K`, RoninKB
slides down from the menubar with NSVisualEffectView vibrancy, the React
UI is already mounted in a WKWebView, and `fetch('/profiles')` lands in
the same process via `roninkb://api/...` — no localhost port, no separate
daemon binary, no browser tab.

The Rust core that today drives `hhkb-daemon` becomes a dylib
(`hhkb-runtime`) linked both into the new `RoninKB.app` and into the
existing `hhkb-daemon` binary (which continues to serve headless / CLI /
Linux / Windows users on `127.0.0.1:7331`). No Rust logic moves; it gets
factored.

The macOS install path stops being "service + browser tab". It becomes
"app you drag to Applications".

## 2. Scope

### In

- New crate `crates/hhkb-runtime/` — Rust dylib exposing a UniFFI
  `Coordinator` interface. Absorbs `AppState`, the `Backend` registry,
  `FlowManager`, `BleManager`, kanata supervisor, and SQLite from
  `hhkb-daemon`.
- `hhkb-daemon` rewritten as a thin axum host that links `hhkb-runtime`
  and wraps its `Coordinator::send` calls in HTTP route handlers.
  External behaviour unchanged.
- New app `apps/hhkb-macos/` — Xcode project (Swift + AppKit) producing
  `RoninKB.app`. Menubar entry point via `NSStatusItem`, panel via
  `NSPanel`, global hotkey via `CGEventTap`, WKWebView for the React UI,
  `WKURLSchemeHandler` for the `roninkb://` scheme.
- `apps/hhkb-app` transport refactor: `daemonClient.ts` gains a
  `Transport` abstraction with two impls — `HttpTransport` (today) and
  `NativeBridgeTransport` (calls `fetch('roninkb://api/...')` and
  subscribes to `window.__roninkb_events`). Decided at module load by
  feature-detecting the bridge global.
- Permission attribution: macOS Input Monitoring, Accessibility, and
  Notifications all prompt under the `gg.solidarity.roninkb` bundle ID
  owned by the app.
- Mutual-exclusion lock file at
  `~/Library/Application Support/RoninKB/runtime.lock`. App and daemon
  both honour it.
- Sparkle-based auto-updater for `RoninKB.app`, EdDSA-signed releases
  published to GitHub releases.
- LaunchAgent rewrite: `gg.solidarity.roninkb.app.plist` runs
  `RoninKB.app/Contents/MacOS/RoninKB --background` at login;
  `gg.solidarity.roninkb.daemon.plist` is removed by the app installer
  on first launch (with user confirmation).
- README rewrite for the macOS install story: "drag to Applications" is
  the headline, `brew install --cask` is the second option, daemon-mode
  install is documented under "Headless".
- Signed, notarized DMG produced by the release workflow.

### Out (explicitly deferred)

- Windows native app (v0.4.0, `apps/hhkb-windows/` with WPF + WebView2).
- Linux native app (no target date; WebKitGTK quirks are out of the
  native-feel skill's evidence base).
- A plugin/extension API.
- Multi-window beyond menubar panel + Settings window + Flow log window.
- Replacing the existing kanata supervisor with in-process kanata.
  kanata stays a child process for LGPL reasons.
- Removing the HTTP API. Daemon mode keeps it exactly as v0.2.0.

### Risk to scope

If M1 reveals that the Cargo + Xcode build loop is slower than ~10 s
incremental, we keep the `hhkb-daemon` HTTP path as the dev-mode default
and ship the native app for end-users only. The architecture is
unchanged; the developer experience just degrades to "build daemon, run
daemon, develop frontend in browser like today" for engineers iterating
on Rust code.

If M2 reveals that `WKURLSchemeHandler` interacts badly with React's
asset loading or with `import()` chunk URLs, we fall back to a localhost
loopback served *inside the app process* (bind `127.0.0.1:0`, get an
ephemeral port, point the WebView at it). One process is still preserved;
the port becomes ephemeral and process-local instead of well-known. RFC
0002 §4.3 covers this fallback.

## 3. Milestones

Each milestone lands as one or more PRs into a long-running
`feature/v0.3.0-native-app` branch. Branch merges to `main` only after
M6. The v0.2.0 release happens first (`feature/v0.2.0-native-backend` →
`main`); v0.3.0 work begins on top.

### M0 — Extract `hhkb-runtime` crate, no behaviour change

- New crate `crates/hhkb-runtime/`. Cargo workspace member. Library
  crate, no binary.
- Move from `hhkb-daemon` into `hhkb-runtime`:
  - `state.rs` → `runtime/state.rs`
  - `backend/*` → `runtime/backend/*`
  - `flow.rs` → `runtime/flow.rs`
  - `kanata.rs` → `runtime/kanata.rs`
  - `kanata_config.rs` → `runtime/kanata_config.rs`
  - `ble.rs` → `runtime/ble.rs`
  - `config.rs` → `runtime/config.rs`
  - `db.rs` → `runtime/db.rs`
  - `error.rs` → split: pure runtime errors in `runtime/error.rs`;
    HTTP-mapping `ApiError` stays in `hhkb-daemon/src/error.rs` and
    `From<RuntimeError>` for `ApiError`.
- `hhkb-daemon` keeps `main.rs`, `routes/*`, `ws.rs`, `ui.rs`,
  `autostart.rs`, `tray.rs`. Route handlers now call into
  `hhkb_runtime::Coordinator` instead of touching state fields directly.
- The kanata `bundled-kanata` feature stays on `hhkb-daemon` (build.rs
  unchanged). The runtime crate accepts a `Option<PathBuf>` for the
  kanata binary at startup; daemon resolves it, app resolves it from its
  `Resources/` dir.
- Tests: every `crates/hhkb-daemon/tests/*` test still passes. Add a
  parallel set of tests in `crates/hhkb-runtime/tests/` that exercises
  the `Coordinator` directly (no HTTP).
- **Exit criteria**: `cargo test --workspace` green. `hhkb-daemon` binary
  byte-identical behaviour from a user POV. `cargo check -p hhkb-runtime`
  passes on macOS / Linux / Windows.

### M1 — UniFFI surface on `hhkb-runtime`

- Add UniFFI 0.28+ as a dependency on `hhkb-runtime`.
- Declare the bridge: `Coordinator` interface, `EventHandler` callback,
  `Request` / `Response` / `RuntimeError` shapes. The exact shape lives
  in `spec/protocol/native-bridge.md` §2–§4; this milestone implements
  it.
- Generate Swift bindings into `apps/hhkb-macos/Generated/Coordinator.swift`
  as part of the Cargo build. A `build.rs` step in `hhkb-runtime` runs
  `uniffi-bindgen swift` against the runtime's `.udl` (or proc-macro
  output) and writes the Swift file.
- Smoke binary: `cargo run -p hhkb-runtime --example coordinator_smoke`
  exercises every endpoint defined in the bridge spec by going through
  the UniFFI layer (the same surface Swift will use).
- **Exit criteria**: `cargo build -p hhkb-runtime` produces
  `libhhkb_runtime.dylib` on macOS, `libhhkb_runtime.so` on Linux,
  `hhkb_runtime.dll` on Windows. UniFFI generates Swift bindings without
  warnings. The smoke binary exercises a profile create/read/update
  cycle.
- **Kill-switch criterion**: if UniFFI 0.28+ proc-macro mode generates
  Swift that doesn't compile against Xcode 16+, fall back to UDL mode
  (UniFFI 0.27) and absorb the small ergonomic loss.

### M2 — Xcode project skeleton, WKWebView, no UI yet

- New Xcode project `apps/hhkb-macos/RoninKB.xcodeproj`.
- Targets: `RoninKB` (app), `RoninKBTests` (XCTest unit), `RoninKBUITests`
  (XCUITest E2E).
- Build phase: a "Run Script" phase that calls `cargo build -p
  hhkb-runtime --release` and copies `libhhkb_runtime.dylib` into the
  app's `Frameworks/`. Incremental builds skip Cargo if the dylib is
  newer than the Rust sources (using `xcrun --find` + mtime check or a
  shipped helper script under `scripts/macos/`).
- Swift code:
  - `App.swift` — `@main` entry, sets `LSUIElement = YES`, creates the
    coordinator.
  - `Coordinator.swift` — bridges `hhkb_runtime::Coordinator` (UniFFI)
    to the rest of the app. Owns `Coordinator::start()` /
    `Coordinator::stop()`.
  - `EventHandler.swift` — implements the UniFFI callback interface;
    enqueues incoming events for delivery to the WebView (M4).
  - `MenubarController.swift` — `NSStatusItem` with template image.
  - `PanelController.swift` — `NSPanel` with `nonactivatingPanel +
    fullSizeContentView` style mask, `NSVisualEffectView` backing,
    placeholder content view.
  - `WebViewController.swift` — `WKWebView` wrapped in `NSViewController`,
    `WKURLSchemeHandler` registered for `roninkb`, loads
    `roninkb://app/index.html` (which is a placeholder HTML file in
    `Resources/web/` for now).
- Permissions plumbing in `Info.plist`: `NSAccessibilityUsageDescription`,
  `NSAppleEventsUsageDescription`, Input Monitoring is requested at
  runtime via `IOHIDCheckAccess`.
- Mutual-exclusion lock file: `RuntimeLock.swift` acquires
  `~/Library/Application Support/RoninKB/runtime.lock` on start, surfaces
  "another instance is running" if held.
- **Exit criteria**: `xcodebuild -project apps/hhkb-macos/RoninKB.xcodeproj
  build` succeeds. Running the app shows a menubar icon. Clicking the
  icon shows an empty `NSPanel` with Aqua vibrancy. The runtime is
  started in-process and reports `Coordinator::get_state()` matches what
  the daemon would have reported. No React UI yet.

### M3 — React UI in WKWebView via `roninkb://`

- `apps/hhkb-app/vite.config.ts`: add a build mode `native-bridge` that
  emits to `apps/hhkb-macos/Resources/web/` instead of `dist/`. The
  emitted bundle uses relative asset URLs so it works under
  `roninkb://app/`.
- `apps/hhkb-app/src/hhkb/daemonClient.ts`:
  - Extract a `Transport` interface: `request(method, path, body?) →
    Promise<Response>`, `subscribe(handler) → unsubscribe`.
  - `HttpTransport` — today's behaviour, used when
    `window.__roninkb_native` is undefined.
  - `NativeBridgeTransport` — used when
    `window.__roninkb_native === true`. Request: `fetch('roninkb://api' +
    path, init)`. Subscribe: register a callback in
    `window.__roninkb_dispatch_table` and have native push events into it
    via injected JS.
  - Module-level `getTransport()` picks one at import time. No runtime
    flag-flipping; the choice is per-bundle.
- Swift `WebViewController` injects a tiny JS bootstrap at
  `WKUserContentController` time: `window.__roninkb_native = true;
  window.__roninkb_dispatch_table = new Set(); window.__roninkb_dispatch
  = (e) => window.__roninkb_dispatch_table.forEach(fn => fn(e));`
- Swift `EventHandler` (from M2) now translates Rust events into JSON,
  calls `webView.evaluateJavaScript("window.__roninkb_dispatch(" +
  jsonString + ")")` on the main queue.
- Build phase ordering: vite's `native-bridge` build runs before
  Xcode's resource copy phase. A `scripts/macos/build-react-bundle.sh`
  invokes `(cd apps/hhkb-app && npm run build -- --mode native-bridge)`.
- **Exit criteria**: launching the app shows the existing React UI
  inside the panel. Every panel in `apps/hhkb-app` works: BindingsPanel,
  KeyDetailPanel, SettingsPanel, BluetoothPanel, FlowBanner. WebSocket
  events arrive (`DeviceConnected`, `KeymapChanged`, etc.) and the UI
  reacts as it does in browser-tab mode.

### M4 — Global hotkey, panel positioning, focus discipline

- `HotkeyManager.swift` — `CGEventTap` listening for `⌘⇧K` (configurable
  in Settings panel, default to `⌘⇧K` because it's RoninKB's letter and
  Raycast already squatted `⌘Space`). The tap is installed at
  `kCGSessionEventTap` and re-armed on disable.
- Panel positioning: `PanelController` computes position from the
  `NSStatusItem`'s frame plus offset, animates open with a 200 ms
  spring; ESC and click-outside dismiss. The panel uses
  `becomesKeyOnlyIfNeeded = true` so focus stays where the user was
  before summoning (Raycast-style "press hotkey while typing in another
  app, type in panel, ESC, you're back in the original app").
- Multi-display: panel always opens on the screen the menubar item is
  on, which is the active screen.
- Tray icon: keep `tray-icon` crate? No. The Swift menubar item replaces
  it on macOS. Remove `tray` feature from `hhkb-daemon` on macOS; keep
  it for Linux/Windows where the daemon is still the user-facing
  surface.
- **Exit criteria**: from any app, pressing `⌘⇧K` summons the panel in
  ≤ 100 ms warm, ≤ 500 ms cold. ESC dismisses. The previous app
  regains focus. Tested on Sequoia and Tahoe.

### M5 — Permission flow, autostart, packaging

- `PermissionsCoordinator.swift` — surfaces required permissions to the
  React UI via a synthetic `roninkb://api/permissions` endpoint:
  - `accessibility` — `AXIsProcessTrustedWithOptions`
  - `inputMonitoring` — `IOHIDCheckAccess`
  - `bluetooth` — `CBManager.authorization`
  - Each entry has `granted: bool`, `openSettingsUrl: String` (deep
    link via `x-apple.systempreferences:com.apple.preference.security`).
- SetupWizard in React already exists (v0.2.0). It now gets a
  `StepNativeApp` step that runs when `__roninkb_native` is true and
  guides the user through the new permission set.
- Autostart:
  - Installer creates
    `~/Library/LaunchAgents/gg.solidarity.roninkb.app.plist` that runs
    `RoninKB.app/Contents/MacOS/RoninKB --background`. The
    `--background` flag tells the app to not open the panel on launch.
  - The same installer detects and removes
    `gg.solidarity.roninkb.daemon.plist` from v0.2.0 if present,
    prompting the user once.
  - A new `Login Items & Extensions` toggle in Settings turns autostart
    on/off via `SMAppService.loginItem(identifier:)`.
- Sparkle integration:
  - Add `Sparkle` framework as an SPM dependency.
  - Generate EdDSA keypair, store public key in `Info.plist`, private
    key in repo secrets.
  - `appcast.xml` published to `https://github.com/kidneyweakx/RoninKB/releases/...`.
  - Release workflow signs the DMG, generates the appcast entry,
    uploads the asset, and publishes the release.
- Sign + notarize:
  - Developer ID Application certificate stored as repo secret.
  - `xcodebuild archive` → `xcodebuild -exportArchive` → `notarytool
    submit` → `stapler staple`.
  - Output: `RoninKB-0.3.0.dmg`.
- Homebrew cask: `roninkb-app` cask that pulls the signed DMG. The
  existing `roninkb` formula (daemon-mode) stays for headless users.
- **Exit criteria**: a fresh-Mac install via the signed DMG:
  - Drag to Applications.
  - First launch: macOS shows zero warnings (notarization holds).
  - SetupWizard guides through Input Monitoring + Accessibility prompts.
  - Login Items shows RoninKB once the user toggles autostart on.
  - Reboot: app launches in background, menubar icon present, panel
    summons via hotkey.
  - Sparkle check finds an update if one is published to the appcast.

## 4. Branch strategy

- Long-running feature branch: `feature/v0.3.0-native-app`. Branched
  from `main` *after* v0.2.0 ships.
- Per-milestone PRs into the feature branch. Each PR runs the full CI
  matrix.
- `main` continues to receive bug fixes for v0.2.x. Periodically rebase
  the feature branch on `main`.
- No fast-forward merge to `main` until M6 exits. Merge-commit
  preferred so per-milestone history is preserved.

## 7. Memory + performance budget

Tenet 4 (perception, not measurement): we measure the things the user
can feel.

| Metric                                       | Budget                | Measurement                                                |
| -------------------------------------------- | --------------------- | ---------------------------------------------------------- |
| Cold launch → menubar icon visible           | ≤ 800 ms              | Instruments + `os_signpost`                                |
| Warm hotkey → panel visible                  | ≤ 100 ms              | XCUITest + `os_signpost`                                   |
| Cold hotkey → panel visible (first summon)   | ≤ 500 ms              | XCUITest + `os_signpost`                                   |
| Resident memory, app idle (panel hidden)     | ≤ 200 MB              | `vmmap --summary`                                          |
| Resident memory, app active (panel visible)  | ≤ 350 MB              | `vmmap --summary`                                          |
| Resident memory, daemon mode                 | ≤ 50 MB (unchanged)   | `ps -o rss`                                                |
| `roninkb://api/profiles` round-trip          | ≤ 5 ms p99            | Coordinator-internal `tracing` span                        |
| `KeymapChanged` event → React state update   | ≤ 16 ms (one frame)   | Instrumented in `daemonClient.ts` and `EventHandler.swift` |

If we miss the resident-memory budget by more than 30%, M5 adds a
margin-cost optimization pass before release (Tenet 8: baseline is what
we communicate; margin is what we attack).
