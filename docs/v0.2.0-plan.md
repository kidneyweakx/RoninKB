## 1. Theme

**Make macOS a first-class platform without forcing Karabiner.** Default install ships a native CGEventTap + IOHIDManager-seize backend that delivers tap-hold, layers, leader keys, and app-aware bindings using only an Apple-gated Input Monitoring + Accessibility prompt ‚ÄĒ no third-party drivers, no signed system extension, no Karabiner. Kanata + Karabiner stays as an opt-in power-user path.

This is a breaking-shape change to `hhkb-daemon` (HTTP API gains `/backend/*`, the old `/kanata/*` becomes a deprecated alias) so it gets a minor-version bump.

## 2. Scope

### In

- New crate `hhkb-macos-native` implementing CGEventTap + IOHIDManager-seize event capture.
- New backend module in `hhkb-daemon` exposing the `Backend` trait for `EEPROM`, `Hidutil`, `MacosNative`, `Kanata`.
- New REST endpoints `/backend/list`, `/backend/select`, `/backend/status` plus `/kanata/*` compat aliases.
- React UI: backend selector in Settings, capability-aware feature gating, first-run wizard rewrite.
- macOS 26 Tahoe support: external keyboards via IOHID seize, built-in via session tap, tested on Tahoe + Sonoma.
- Tap-hold + layers + leader + hyper + app-aware bindings via `keyberon` (or `smart-keymap`, decided in M1).
- LaunchAgent for `Hidutil` re-application at login.
- Documentation: README rewrite for the new install story, RFC kept current, RELEASING.md updated for the multi-backend artefact set.

### Out (explicitly deferred)

- Macros / multi-key sequences on the native backend (kanata path remains for these in v0.2.0).
- Per-keyboard rules on the native backend (CGEventPost re-injection loses source identity ‚ÄĒ kanata path only).
- Rebuilding the entire kanata feature surface (no point ‚ÄĒ keep it as the power-user backend).
- Linux / Windows backend changes ‚ÄĒ they continue using kanata as default, no change.
- HHKB hardware EEPROM protocol expansion ‚ÄĒ continues with current 128-byte / 2-layer model.

### Risk to scope

If the M1 PoC reveals that CGEventTap-based tap-hold is unworkable even with our chosen tunings, we down-scope: ship with EEPROM + Hidutil as macOS defaults and keep advanced bindings on the kanata path. RFC is unchanged; we just don't promise tap-hold on the native backend in v0.2.0 and revisit in v0.3.

## 3. Milestones

Each milestone lands as one or more PRs into a long-running `feature/v0.2.0-native-backend` branch. Branch merges to `main` only after M5.

### M0 ‚ÄĒ Backend trait + capability bits ‚ÄĒ small, no behaviour change

- Define `Backend` trait, `Capabilities`, `PermissionStatus`, `BackendError` in a new `crates/hhkb-daemon/src/backend/mod.rs`.
- Wrap existing kanata supervisor as `KanataBackend` impl. Wrap EEPROM / `hhkb-core` device as `EepromBackend` impl.
- Daemon owns `Vec<Box<dyn Backend>>` + an `active: Option<Box<dyn Backend>>`. For M0, behaviour is identical to v0.1.x: kanata is auto-selected on every platform exactly as today.
- Tests: existing kanata tests still pass; new trait-conformance tests for both backends.
- **Exit criteria**: `cargo test --workspace` green, daemon behaviour unchanged from user POV.

### M1 ‚ÄĒ macOS native backend PoC

- New crate `crates/hhkb-macos-native`. Cfg-gated `target_os = "macos"`.
- Spike: open a `CGEventTap`, log every key press for 30 seconds, exit. Confirm the tap callback fires under modern macOS code signing.
- Spike: open `IOHIDManager` with seize on a real external keyboard, decode HID reports. Confirm we can intercept on Tahoe.
- Spike: feed a synthetic event sequence into `keyberon::layout::Layout` configured for a simple tap-hold (Esc tap / Ctrl hold on Caps), verify the engine output matches expectation.
- Decision point: pick `keyberon` vs `smart-keymap` based on PoC ergonomics. Document the decision inline in `crates/hhkb-macos-native/README.md`.
- **Exit criteria**: a runnable example binary that intercepts a real keyboard, runs through the engine, posts re-injected events, and demonstrably swaps Caps‚ÜíCtrl on tap-hold against a live Mac.
- **Kill-switch criterion**: if tap-hold latency on the spike exceeds 300ms or fast typing makes home-row mods misfire >20%, abort the native backend for v0.2.0 and re-scope (see ¬ß2 risk).

### M2 ‚ÄĒ `hhkb-macos-native` integrated as a `Backend` impl

- Wire `MacosNativeBackend` into `hhkb-daemon`. Profile JSON ‚Üí engine layout translation in `backend/macos_native/profile.rs`.
- Implement `permission_status()` checks: `CGPreflightListenEventAccess`, `AXIsProcessTrustedWithOptions`, IOHIDManager open success.
- Implement `apply` / `teardown` / `reload`. Reload uses in-process state swap (drop the old layout, install the new one) ‚ÄĒ no process restart, no dropped modifiers.
- App-aware binding via `NSWorkspaceDidActivateApplicationNotification` (event-driven, not polled).
- **Exit criteria**: end-to-end on macOS Sequoia and Tahoe ‚ÄĒ daemon applies a profile that swaps Caps to Ctrl/Esc tap-hold, types correctly under normal load, hot-reloads when the profile changes.

### M3 ‚ÄĒ `Hidutil` backend + LaunchAgent

- New `backend/hidutil.rs`. Generates `UserKeyMapping` JSON from the per-key portion of the profile.
- LaunchAgent management ‚ÄĒ write `~/Library/LaunchAgents/gg.solidarity.roninkb.hidutil.plist` that re-applies on login.
- Handle macOS-version-dependent breakage: detect missing `property` verb on Tahoe betas, downgrade gracefully.
- **Exit criteria**: a Caps‚ÜĒEsc swap survives reboot without the daemon running, and applies to every USB keyboard the user plugs in.

### M4 ‚ÄĒ Backend selection, REST API, frontend rewrite

- `GET /backend/list`, `POST /backend/select`, `GET /backend/status` ‚ÄĒ implemented and tested via existing route-test infra.
- `/kanata/*` compat: identical behaviour to v0.1.x when the kanata backend is active; returns clear `409 backend_inactive` JSON when a different backend is selected.
- React: replace the kanata-specific section in `SettingsPanel.tsx` with a unified "Software bindings" pane. Add a backend selector (default = auto, Advanced = pin to specific). Capability-aware feature gating in `BindingsPanel.tsx`.
- First-run wizard rewrite ‚ÄĒ detects platform, picks the right backend story, deep-links to the right System Settings pane for missing permissions.
- **Exit criteria**: user-visible flow on macOS ‚ÄĒ fresh install ‚Üí wizard ‚Üí grant Input Monitoring ‚Üí wizard verifies a tap-hold works ‚Üí user lands on the bindings editor with a working Caps‚ÜíCtrl/Esc binding. No mention of Karabiner unless they open Advanced.

### M5 ‚ÄĒ Documentation, packaging, release

- README rewritten: install command stays the same; the macOS section drops the "must install Karabiner" caveats and gains a "Power users ‚ÄĒ install Karabiner-Elements for sub-100ms tap-hold" footnote.
- RELEASING.md updated for the new artefact shape (`hhkb-macos-native` builds into `hhkb-daemon` via cfg, no separate binary).
- CLAUDE.md updated for new feature flags and macOS-specific dev paths.
- Formula: drop the kanata + Karabiner caveats from the macOS section; the kanata backend is now opt-in via UI.
- **Exit criteria**: `feature/v0.2.0-native-backend` merges to `main`, `v0.2.0` tag pushed, release workflow ships cleanly, post-install validation per RELEASING.md ¬ß4 succeeds on Sequoia + Tahoe + Linux x86_64.

## 4. Branch strategy

- Long-running feature branch: `feature/v0.2.0-native-backend`. Per-milestone PRs into it; each PR runs the full CI matrix from `.github/workflows/ci.yml`.
- `main` continues to receive bug fixes for v0.1.x (which would ship as v0.1.2, v0.1.3 etc.). Periodically rebase the feature branch on `main`.
- No fast-forward merge to `main` until M5 exits cleanly. Squash merge or merge-commit acceptable; preference for merge-commit so the per-milestone history is preserved.

## 5. Test plan

### Per-backend unit tests

- `EepromBackend`: re-uses existing `hhkb-core` device tests, plus new conformance test against the `Backend` trait.
- `HidutilBackend`: parser tests for `UserKeyMapping` generation; integration test (gated, run-on-mac-only) that sets a known mapping, reads it back via `hidutil property --get`, restores.
- `MacosNativeBackend`: engine-translation tests in CI (no event injection); manual integration test in M1 / M2 / M4 against real hardware.
- `KanataBackend`: existing kanata tests; new conformance test.

### End-to-end acceptance ‚ÄĒ must pass on every backend before release

- Apply a profile with one binding (Caps ‚Üí Ctrl/Esc tap-hold).
- Type a known sequence; assert output matches expectation.
- Hot-reload to a different profile (Caps ‚Üí Esc plain).
- Tear down; assert no leftover state in TCC, LaunchAgents, or running processes.

### Tahoe-specific acceptance

- External USB keyboard on Tahoe: native backend captures via IOHID seize, applies the profile.
- Built-in keyboard on Tahoe: native backend captures via session tap.
- Hotplug: external keyboard removed and re-attached during a session ‚ÄĒ backend re-seizes without daemon restart.

### Compat acceptance

- `/kanata/*` endpoints return identical responses on macOS when the kanata backend is active (proves backwards compat with any v0.1.x clients).
- Existing user profiles from v0.1.x parse and apply via the native backend without modification.

### CI matrix

Same 3-OS matrix as v0.1.x (`.github/workflows/ci.yml`). Adds:

- `cargo check -p hhkb-macos-native` on macOS only.
- `cargo check -p hhkb-daemon --features backend-native,backend-hidutil,backend-kanata` (all three concurrently).
- Frontend build matrix unchanged.

| R4  | Code signing / TCC silent disable race (see Daniel Raffel Feb 2026)   | Medium     | Medium | Stable Developer ID signing + entitlements review at M5 |

| R7  | macOS / Tahoe sysext rules change again before ship                   | Low        | High   | Track Apple Developer release notes through M2-M5; prepare to re-spike if needed |

## 8. Rollback plan

If a critical post-release issue lands, rollback is simple because v0.2.0 is feature-additive on every backend except the macOS default switch:

- **Hotfix**: revert the default-backend selection logic to prefer kanata-on-Karabiner when present, ship as v0.2.1. Native backend stays available as opt-in.
- **Full rollback**: tag a `v0.1.x` patch release that restores the v0.1.x install story. Users `brew upgrade roninkb@0.1` (or just `brew install roninkb` if we don't keep two formulae).
- The native backend code itself is feature-gated; disabling `backend-native` at compile time gives a v0.1.x-equivalent binary.

Kill switch in production: `~/Library/Application Support/roninKB/config.toml` accepts `[backend].pin = "kanata"` to override automatic selection ‚ÄĒ users can pin themselves out of the native backend without waiting for a daemon update.
